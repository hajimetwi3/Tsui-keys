# Tsui keys

ブラウザで完結するローカルパスワードマネージャー。外部送信なし、アカウントなし、同期なし。

> Cloudflare Pages 経由でアクセスする場合、初回アクセス時とアプリ更新時に Cloudflare からの通信が発生します。アプリ起動後、本アプリ本体が vault 内容・入力内容を外部送信することはありません。
> なお、作者プロフィールや Tsui series のランディングページ等の情報ページでは、訪問数の把握に Cloudflare Web Analytics を利用しています（Cookie なし、フィンガープリントなし、クロスサイトトラッキングなし）。

[**Tsui series**](https://hajimetwi3.github.io/hajimetwi3/Tsui-series/)の一つです。

## 特徴

### 思想

- **完全ローカル動作**: CSP `connect-src 'none'` により、ブラウザレベルで外部送信を禁止
- **外部モジュール依存なし**: CDN も npm もなし。ブラウザ標準の WebCrypto API のみ使用
- **隠れた挙動なし**: 自動保存しない、自動同期しない、本体には計測機能を入れない
- **「効くか効かないか不明な機能」は搭載しない**: 偽の安心感を避けるため

### 仕様

- **独自ファイル形式 `.tkey`**: AES-256-GCM + PBKDF2-SHA256 (800,000 iterations)
- **単一 HTML 配布**: ダウンロードして `file://` で開くだけ
- **無操作で自動ロック**: 既定 5 分(設定可)
- **多言語**: 日本語 / English(下部パネルから切替)
- **PWA 対応**: Cloudflare Pages での `https://` 配信時にホーム画面追加可

## 設計原則

### 「効くか効かないか不明な機能」は搭載しない

ブラウザ仕様や環境に依存して、効いたり効かなかったりする種類の機能は、ユーザーに偽の安心感を与えるため搭載しません。たとえば以下の機能は意図的に実装していません。

- **クリップボード自動クリア**: タブ非アクティブ時に確実な自動クリアは可能ではありません。代わりに、ヘッダの「CB消去」ボタンによる手動上書きを提供します(上書き文字列は `(cleared by Tsui keys)`)。
- **ファイル排他制御（ロックファイル方式）**: タブを閉じる瞬間のクリーンアップが保証されないため、誤ったロック状態が残り続けるリスクの方が大きい。複数タブで同じ vault を開かない運用をお願いします。

### 認証要素を同じ箱に入れない

パスワードと TOTP（2要素認証コード）を同じ vault に集約すると、片方が漏れた時にもう片方も同時に漏れる構造になります。本アプリは TOTP 生成を同梱していません。TOTP は別のアプリやハードウェアキーに委ねることを推奨します。

### 外部モジュール依存ゼロ

CDN も npm も使いません。WebCrypto API のみで暗号処理を完結させます。

### 隠れた挙動なし

自動保存しない、自動同期しない、本体には計測機能を入れない。保存・同期はすべてユーザーの明示的な操作（ファイル保存ボタンや Ctrl+S）で発生します。

なお、作者プロフィールや Tsui series のランディングページ等の情報ページでは、訪問数の把握に Cloudflare Web Analytics を利用しています（Cookie なし、フィンガープリントなし、クロスサイトトラッキングなし）。本アプリ本体（`tsui-keys.html`）では一切利用しておらず、CSP `connect-src 'none'` により、計測コードを埋め込んでもブラウザの CSP 機構で外部送信が禁止される構造です。

ブラウザの永続化領域には以下を保存しています:

- **localStorage**(UI 設定のみ)
  - `tsui-keys:sidebar-w` ... サイドバー幅(px)
  - `tsui-keys:sidebar-h` ... 縦積み時のサイドバー高さ(px)
  - `tsui-keys:panel-open` ... 下部パネル開閉状態('1' / '0')
- **IndexedDB**(前回の vault ファイルへの再アクセス用の情報のみ)
  - DB: `tsui-keys` / store: `recent` / key: `lastVault`
  - 値: ファイル名とハンドル(`FileSystemFileHandle`)
  - 再アクセス時はユーザー操作を起点に permission を確認し、未許可の場合のみブラウザに permission を要求します(自動アンロックはしません)

**マスターパスワード・vault の中身・エントリは、上記いずれにも保存しません**（vault ファイル `.tkey` の中にのみ存在し、ロック / タブクローズ時にメモリからも破棄されます）。

## CSV インポート

vault menu の「インポート」から、Tsui keys の JSON または CSV ファイルを取り込めます。CSV は次の 2 形式に対応します。列名の大文字小文字や前後の空白は無視されます。

### Tsui keys CSV 形式（推奨）

```csv
title,username,password,url,notes,tags
GitHub,hajime,p@ss,https://github.com,業務,work;dev
銀行,user01,123456,https://bank.example,,finance
"備考に, カンマ含み",user,pw,,"複数行も
OK","tag1;tag2"
```

- 列順は任意。`title` または `name` 列が必須。`password` 列は任意
- `tags` は `;` 区切り（`,` も許容）。各タグは最大 32 文字、1 エントリあたり最大 32 個
- セル内のカンマや改行はダブルクオート `"..."` で囲む。クオート自体は `""` でエスケープ
- BOM 付き UTF-8、CRLF / LF どちらの行終端にも対応

### Chrome / Edge エクスポート形式

ブラウザの「パスワードのエクスポート」で出力される CSV をそのまま取り込めます。

```csv
name,url,username,password,note
GitHub,https://github.com,hajime,p@ss,
```

`name` 列が `title` に、`note` 列が `notes` にマップされます。`tags` は付きません（必要なら取り込み後に編集してください）。

### その他のツールから移行する場合

他のパスワードマネージャーから出力された CSV は形式が異なります。`title` または `name` 列さえ認識できれば部分的に取り込めますが（`password` 列は任意）、確実にしたいときは表計算ソフト等で列を `title, username, password, url, notes, tags` に並べ替えてから取り込んでください。

### 取り込み挙動

- **追加（append）**：既存エントリを残し、新しい ID で追加
- **置き換え（replace）**：既存エントリをすべて削除してから取り込む
- ID は自動再採番されるので、同じ vault に同じファイルを 2 回取り込むと重複します（必要なら手で整理）
- パース失敗行はスキップして件数を表示。1 行の不備で全体が止まることはありません

## vault のマージ（複数端末向け）

別々の端末で編集した 2 つの `.tkey` ファイル（A と B）を合成し、新しい vault を作る機能です。起動画面の「2 つの vault をマージ」ボタンから起動します。

### フロー

1. **A の `.tkey` を選択** → A のマスターパスワードを入力
2. **B の `.tkey` を選択** → B のマスターパスワードを入力（A と異なるパスワードでも可）
3. **マージ方法を選ぶ**
   - **ID で照合**：エントリの内部 ID が一致するものを「同じ」とみなす（同一 vault のコピーを両端末で編集したケース向け）
   - **名前（title）で照合**：title を trim + 大小無視で一致するものを「同じ」とみなす（A と B で別々に同じサービスを登録したケース向け）
   - **新しい方を採用（preferNewer）**：ON なら updatedAt が新しい方を採用、OFF なら両方を残す
4. **プレビュー**：「計 N 件（A 単独 / B 単独 / 自動マージ / 重複保持）」を表示
5. **新 vault のマスターパスワードを決めて保存** → 保存後そのまま新 vault を開いた状態になる

### 設計上の判断

- **A と B のファイルは変更しません**（純関数的なマージ、元データは無傷）
- **削除は反映されません**（tombstone を持たない設計のため、片方で削除したエントリは新 vault に残ります。後で手動整理してください）
- **新 vault のエントリ ID は全て新規採番**（マージ前の ID を引き継がない）
- **3 つの vault のマスターパスワードは独立**（A・B・新 vault それぞれ別でも構いません）
- マージ結果が意図と違った場合は、vault menu の「危険な操作」→「この vault ファイルを削除」で破棄できます

### よくあるユースケース

| ケース | 推奨設定 |
|---|---|
| メイン PC の vault をコピーしてサブ機で編集 → マージしたい | ID で照合 + 新しい方を採用 |
| メイン PC とサブ機で別々に同じサービスを登録 → 統合したい | 名前で照合 + 新しい方を採用 |
| 慎重にコンフリクト確認したい | どちらかのモード + preferNewer OFF（両方残してから手動整理）|

## マスターパスワード

vault のセキュリティは、最終的にマスターパスワードの強度に大きく依存します。AES-256-GCM + PBKDF2-SHA256（800,000 iterations）でファイルが守られていても、マスターパスワード自体が短く単純なら攻撃者の総当たりで破られます。

「弱いマスターパスワードでも、ツールの強力なアルゴリズムが守ってくれる」という発想には限界があります。短い・推測されやすいマスターパスワードは、どんな KDF を使っても比較的短時間で破られます。**ツール側で攻撃コストを高めることはできますが、最終的な耐性のボトルネックはマスターパスワード自体の強度**になります。

## 配布形態と推奨運用

Tsui keys は、単一 HTML 配布版（GitHub Releases）と Cloudflare Pages 配信版の 2 通りで利用できます。**用途に応じて使い分けてください**。

### 第一推奨：ダウンロード + `file://` 起動

`tsui-keys.html` をダウンロードしてローカルに保存し、ファイルをダブルクリック（またはブラウザで `file://` で開く）。

| 利点 | 説明 |
|---|---|
| ネットワーク経路ゼロ | `connect-src 'none'` と組み合わさり、外部送信が CSP によりブロックされる |
| 配信プロセスからの独立 | 一度ダウンロードすれば、配信元の改ざんを受けない |
| オフライン保証 | ネットワーク接続なしで完全動作 |
| 拡張機能のリスク低減 | 主要なブラウザは `file://` への拡張アクセスを既定で無効化しているが、ユーザー設定で個別許可されている場合もある。確実に拡張機能の影響を排除したい場合は、別途プロファイル分離(後述)を併用する |
| 更新タイミングの完全制御 | 勝手にアップデートされない |

**本格利用にはこちらを強く推奨**します。

### 第二推奨：HTTPS Web 配信（PWA）

スマホやお試し用、出先利用などのために、Cloudflare Pages 経由で公開された `https://` でも利用できます。本アプリは PWA（Progressive Web App）として動作するため、ブラウザの「ホーム画面に追加」機能でアプリ風に起動可能です。

| 利点 | 説明 |
|---|---|
| スマホでの利用 | Android Chrome の `content://` 問題を回避 |
| インストール不要のお試し | URL アクセスだけで使える |
| ホーム画面追加 | 一度キャッシュされれば、オフラインでも起動可能 |

**注意点 — 配信元改ざんのリスク**

PWA 経由の利用では、サーバー侵害や配信経路上での改ざんなどで配信ファイルが書き換わった場合に、ユーザー側で配信物の整合性を検証する標準的な動線がありません（ブラウザキャッシュや Service Worker 経由で取得された bytes は、リリース時のハッシュと照合する手段が事実上なく、PWA の自動更新タイミングも見えにくいため）。

リスクを下げるために、以下を推奨します:

- **重要な vault を扱う際は、第一推奨の `file://` 起動を選ぶ**。ダウンロードしたファイルはハッシュ照合で正規性を確認できる
- **配信者の公開発信を確認しておく(X 等)**。リリース告知や注意喚起を追えるようにしておく
- **普段と異なる挙動（起動が遅い、見たことのないダイアログ、想定外の挙動）があれば、PWA をアンインストールして再アクセス**し、変化を確認する

### スマホ運用について

現時点で、スマホ単体での `.tkey` 編集は**多くの環境で制約があります**：

- iOS Safari は File System Access API 未対応 + `file://` 起動が事実上不可能
- Android Chrome は `content://` 経由で開くと FSA が SecurityError になる可能性あり。

**実用的な運用パターン**：

- PC で `.tkey` を編集 → USB ケーブルでスマホに転送 → スマホは閲覧用として PWA で `https://` 配信を使う
- スマホでも本格的に編集したい場合、Android Chrome の比較的新しいバージョンで `https://` 配信を経由する運用が候補

なお、Android Chrome の一部バージョンでは、ローカルにダウンロードした `tsui-keys.html` から **vault の作成・編集まで動作する**ケースが観察されています（Android Chrome の FSA モバイル対応の進展による）。ただし、ローカル起動時はクリップボードへのコピー操作が動作しない等の制約も観察されており、こうした挙動はブラウザの実装に依存するため、**本アプリとしては動作保証しません**。動けば便利機能、安定運用したい場合は PC での `file://` 起動が第一推奨という位置付けは変わりません。

## 推奨される運用

ブラウザ環境で動作する以上、いくつかの運用上の心がけで安全性を上げられます。

### 推奨

- **配布物（`tsui-keys.html`）を端末にダウンロードし、`file://` で開く**。これは外部送信や拡張機能の影響を低減しやすい使い方です（外部送信、拡張機能のデフォルトアクセス、各種統計収集などの大半が止まります）
- **重要な vault を開く際は、拡張機能を入れていない「専用ブラウザプロファイル」の利用を検討**してください。多くのブラウザは複数プロファイルを切り替えられます
- **マスターパスワードはこのツール以外で使わない**。他サービスとの使い回しは避けてください（マスターパスワードが他サービスで漏れると、vault も連鎖的に脅かされます）
- **`.tkey` ファイルは利用者の責任でバックアップ**してください

### 避けてほしい操作

- **画面内容の自動翻訳・右クリックからの翻訳**：本アプリは翻訳除外を指示していますが、明示的な操作まではブラウザ側で止められません。誤って翻訳メニューを選択すると、表示中の内容が外部サービスに送信される可能性があります
- **ブラウザ内蔵 AI / 補助機能による「ページ要約」「内容について質問」など**：同様の理由で、操作した瞬間にデータが外部に行く場合があります
- **複数タブで同じ `.tkey` を同時に開く**：排他制御は行っていないので、最後に保存したタブの内容で上書きされます
- **共有 PC やインターネットカフェ等での利用**：マスターパスワードのキー入力やメモリ残留のリスクが上がります

### 内部での扱い

vault のロック後は、メモリ上で保持しているマスターパスワード・平文エントリへの参照を破棄します。暗号化・復号時に生成される一部の平文バイト列は best-effort でゼロ埋めします。ただし、JavaScript 文字列・オブジェクト・WebCrypto 内部鍵の完全な即時消去はブラウザ仕様上保証できません。**機微な vault を扱った後は、ブラウザを終了するとメモリの早期解放を促せる可能性があります**(OS の挙動に依存するため完全な消去は保証されません)。

## バックアップ

vault ファイル（`.tkey`）のバックアップは**利用者の責任**で取ってください。本アプリは自動バックアップを行いません。マスターパスワードを忘れた場合や、ファイルが破損・消失した場合、アプリ側で復旧する手段はありません。

### 推奨される取り方

普段使う `.tkey` とは**別の安全な場所**に定期的にコピーすることを推奨します。例：

- 外付け SSD、USB メモリ
- 暗号化された外部メディア
- 別の PC

### `.tkey` を外部に保管する場合

`.tkey` ファイルは**それ自体が AES-256-GCM で暗号化**されています。マスターパスワードが十分に強ければ、ファイルがコピーされたり外部に流出しても、ファイル自体から内容を読み取られるリスクは低く保たれます（マスターパスワードの強度に依存します）。

ただし、**マスターパスワードを忘れたら復旧不能**という性質は変わりません。マスターパスワードは：

- 物理的なメモ（紙、金庫）
- ハードウェアパスワードマネージャー
- 別系統の暗号化ストレージ

など、`.tkey` とは別の場所にも保管しておくと、単一障害点を減らせます。

## ファイル形式 .tkey (version 1)

```
offset  len  field      value
   0     4   magic      "TKEY" (0x54 0x4B 0x45 0x59)
   4     2   version    BE uint16 = 0x0001
   6     1   kdf        0x01 = PBKDF2-SHA256, 0x02 = PBKDF2-SHA512
                        0x03〜 は将来の拡張用予約
   7     4   iter       BE uint32 (PBKDF2 iterations, 100000〜10000000)
  11    32   salt       CSPRNG 生成(保存毎に再生成)
  43    12   iv         AES-GCM nonce(保存毎に再生成)
  55     N   cipher     AES-256-GCM(平文 JSON、末尾 16B tag)
```

ヘッダ 55 バイトは GCM の **AAD** に含まれるため、ヘッダ改ざんも認証エラーで検出されます。

平文 payload は UTF-8 JSON:

```json
{
  "schema": 1,
  "createdAt": "2026-04-26T03:00:00.000Z",
  "updatedAt": "2026-04-26T03:05:00.000Z",
  "settings": { "keepPasswordHistory": true },
  "entries": [
    {
      "id": "uuid-v4",
      "title": "GitHub",
      "username": "user@example.com",
      "password": "...",
      "url": "https://github.com",
      "notes": "...",
      "tags": ["work", "dev"],
      "previousPassword": null,
      "createdAt": "...",
      "updatedAt": "..."
    }
  ]
}
```

`kdfId` は将来の KDF 拡張用に予約フィールドを持ちます。新しい kdfId が割り当てられた場合も、既存 vault は元の kdfId のまま復号できる後方互換設計です。

## 暗号アルゴリズム

| 役割 | アルゴリズム |
|---|---|
| 対称暗号 | AES-256-GCM（NIST 認定、IETF 標準、AEAD、AES-NI 支援、WebCrypto ネイティブ） |
| KDF | PBKDF2-HMAC-SHA-256, 800,000 iterations。PBKDF2 採用時の目安として、現時点(2026年4月)で OWASP が示す 600,000 iterations 以上の水準を満たす設定です。 |
| 乱数 | `crypto.getRandomValues()`（OS のエントロピープールから取得する CSPRNG） |
| 完全性 | AES-GCM の認証タグ + ヘッダ全体を AAD に含める |

### 対抗できない脅威

以下等は本アプリの範疇外です:

- OS レベルのマルウェア・キーロガー
- 物理的に端末にアクセスできる攻撃者
- ブラウザ拡張機能（`file://` で最小化可）
- ブラウザや OS 自体の重大な脆弱性
